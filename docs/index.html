Bank Account Management System

A C++ object-oriented console application that simulates a simple banking system with user accounts, transaction logging, and manager-level administration tools.

Authors: Jack London, Ethan Christman, David Desing
Course: CompSci 371 – Final Project

Language: C++ Paradigm: OOP Persistence: Text Files Modules: User · BankAccount · Manager · Transactions

At a Glance

  • User account creation & login
  • Checking & Saving account types
  • Deposits, withdrawals, balance viewing
  • Transaction history with timestamps
  • Manager panel: view & delete users, view history

System Architecture

The project follows a modular object-oriented design. Each major responsibility is encapsulated in its own class or module.

Core Classes

  • User – Represents a bank customer and handles loading/saving to Users/<username>.txt.
  • BankAccount – Abstract base class exposing common operations like deposit, withdraw, and the account menu.
  • CheckingAccount / SavingAccount – Implement account-specific behavior via polymorphism.

Supporting Modules

  • Transactions – Parses and prints transaction history; adds initial deposit entries.
  • Deposit / Withdraw – Encapsulate the actual balance updates and transaction logging.
  • Menu – Simple textual menu renderer reused for main menu, account menu, and manager menu.
  • Manager – Namespace that implements manager login and admin tools.

UML (Conceptual)

High-level class relationships:

User <-- BankAccount* (composition at runtime) BankAccount (abstract) ├─ CheckingAccount └─ SavingAccount User ├─ owns: username, password, balance, accountType, transactions └─ persists to: Users/<username>.txt Manager (namespace) ├─ ManagerLogin() ├─ managerMenu() ├─ ViewAUser() └─ ViewUserHistory() Transactions (namespace) ├─ parse(history) ├─ print(User*) └─ addInitialDeposit(User*, amount)

Implementation Highlights

The implementation is focused on clear data flow, safe input handling, and readable modular code.

User Creation Flow

  • Prompt for first and last name.
  • Validate username (no spaces, not already taken).
  • Prompt for password.
  • Choose account type: Checking or Saving.
  • Enter initial deposit (minimum of $20, or cancel with -1).
  • Save all data to Users/<username>.txt.
  • Record an INITIAL_DEPOSIT transaction.

Login & Account Menu

  • Login searches for the user’s file and verifies the Password: line.
  • On success, constructs the appropriate account type:
    CheckingAccount SavingAccount
  • Account menu options:
    • Deposit
    • Withdraw
    • Account Info
    • View Activity
    • Log Out

File Format Example

Account#: 1002 Username: desidj27 Password: taco First: David Last: Desing Balance: 722.03 Account Type: Checking Transaction History: INITIAL_DEPOSIT:300.12 TIME:2025/12/10 17:17:10 NEW_AMOUNT:300.12|DEPOSIT:421.91 TIME:2025/12/10 17:17:44 NEW_AMOUNT:722.03|

This line-oriented format makes it straightforward to parse user records and transaction history when loading from disk.

Transaction System

All balance-changing actions are logged using a unified text format so that the history can be parsed and displayed later.

Log Format

Each entry uses the following pattern:

TYPE:amount TIME:YYYY/MM/DD HH:MM:SS NEW_AMOUNT:balance_after|
  • TYPE ∈ {INITIAL_DEPOSIT, DEPOSIT, WITHDRAW}
  • amount – the transaction amount
  • TIME – timestamp when the action occurred
  • NEW_AMOUNT – resulting account balance

Parsing and Display

  • Transactions::parse splits the history on |.
  • Each entry is parsed into a Transaction struct with:
    • type
    • amount
    • time
    • newAmount
  • Transactions::print(user) prints a formatted table for:
    • User account menu (View Activity)
    • Manager panel (View User’s History)

Manager Tools

A separate Manager login provides administrative capabilities, including viewing users and their histories and deleting accounts.

Manager Login

  • Manager credentials are currently hard-coded as admin / password.
  • On success, the manager is shown a dedicated menu.

In a future version, managers could be stored in a separate secure file or database.

Manager Menu Options

  • View All Users
  • View A User (summary info)
  • View A User's Transaction History
  • Delete A User
  • Delete All Users
  • Logout

Testing & Validation

Testing focused on boundary conditions, file persistence, and correctness of transaction histories.

What We Tested

  • Creation of users with valid and invalid input.
  • Login failures: wrong username and wrong password.
  • Deposit and withdraw with edge cases:
    • Negative amounts
    • Zero amounts
    • Withdraw > balance
  • Transaction log updates after each operation.
  • Manager tools: viewing, deleting, and listing users.

Future Enhancements

  • Encrypt stored passwords and user data.
  • Add a graphical user interface (GUI).
  • Support multiple accounts per user (e.g., checking + savings + credit).
  • Move from text files to a relational database.
  • Introduce transaction IDs and richer metadata.
  • Enable networked multi-user access.
